Welcome

MetalFury is both a philosophy and a practical engine implementation. The implementation lives under the MetalFury::Core namespace and expresses the MetalFury Manifesto through a modular, test-first architecture. This documentation highlights the core architectural concepts, the systems model, and how the engine exposes APIs for applications, editors, and plugins.

Architecture Overview

Though concise here, the architecture is documented in detail throughout the API reference under the MetalFury::Core namespace. The high-level goals are:

  • Explicit, opt-in dependencies
  • Small, well-tested, replaceable systems
  • Clear separation between Engine and Application
  • Editor-as-first-class-citizen for rapid iteration

Systems Model

At the heart of MetalFury is the concept of a "system". A system is a self-contained module with a single, well-defined responsibility. Systems are composed and wired together by the application. Two primary categories exist:

Core Systems

Core systems live in MetalFury::Core::Systems and manage global concerns that are independent of entities. Typical responsibilities include:

  • Rendering (Renderer, SwapChain, MaterialManagers)
  • Audio output (AudioDevice, Mixer)
  • Physics simulation (PhysicsWorld, CollisionManager)
  • Input processing (InputManager)
  • File I/O and configuration (ConfigurationManager)
  • Scripting interfaces (ScriptingEngine)

Core systems are often created and registered at engine startup and provided to other systems via explicit dependency injection.

ECS Systems

ECS systems operate within the MetalFury ECS world and are responsible only for processing entity component data. Examples:

  • MovementSystem
  • DamageSystem
  • HealthSystem
  • AnimationSystem
  • TargetingSystem

ECS systems declare the component archetypes they care about and never reach outside the ECS world for global state. If an ECS system requires access to a core service, it must be passed that dependency by the application.

Core System ↔ ECS System Interaction

Core systems and ECS systems are deliberately decoupled. Interactions between these layers are achieved through two primary patterns:

  • Dependency injection (application wires a core system into an ECS system)
  • Event-driven communication (systems publish and subscribe to lightweight events)

Example (conceptual):

// At application composition time
auto renderer = std::make_shared<MetalFury::Core::Systems::Renderer>();
auto renderSystem = std::make_unique<RenderECSSystem>(renderer);
ecs.registerSystem(std::move(renderSystem));

Event and Action Architecture

MetalFury coordinates runtime behavior with a dual-model:

Events (instant)

Events are immediate, synchronous broadcast messages used for "happened now" semantics (collisions, input, state changes). They follow a publish/subscribe pattern and are type-safe. Event handlers are invoked in the frame where the event is fired.

Actions (deferred)

Actions are explicit, deferred units of work submitted to an Action Queue. They encapsulate a callable and parameters and may return results. Actions are used for scheduling work outside the immediate frame or delegating tasks between systems when the requester is allowed to access the target system.

Important design rule:

  • The requester must already have access to the system it delegates an action to.

Editor and Workflow

The MetalFury Editor is designed as the first game produced by the engine. Editor responsibilities include:

  • Creating a project workspace and boilerplate
  • Compiling code for Editor Mode (hot-loadable DLL) and Release Mode
  • Managing hot reload and workspace lifecycle

This model enables live editing and rapid iteration while preserving a clean separation between tooling and runtime.

Scripting (Lua)

Lua is the default scripting language for MetalFury. Scripting binds into the core engine through stable, sandboxed APIs. Lua is used to define game behaviour, hook into systems, and drive iteration workflows.

Plugins and Dynamic Extension

Plugins are supported as dynamic runtime extensions. Key points:

  • Plugins are loaded after core systems initialize
  • Plugins register via a defined interface and may optionally expose headers and libs during development
  • At runtime a plugin DLL is sufficient

Plugins allow modular feature addition without altering the engine core.

Engine ↔ Application Contract

MetalFury enforces a two-part contract:

  • The Engine provides a reusable, testable library (MetalFury::Core)
  • The Application orchestrates systems, composes dependencies, and defines gameplay

This separation keeps the engine flexible, non-prescriptive, and suitable for many kinds of projects.
